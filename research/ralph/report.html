What is Ralph Wiggum's Loop?

The Ralph Wiggum Technique (also known as "Ralph's Loop" or simply "Ralph") is an AI development methodology built around continuous AI agent loops. Created by Geoffrey Huntley, this technique went viral in late 2025 and has since become one of the most influential patterns in autonomous AI-assisted coding.

In essence: Ralph is a Bash loop. A simple while true that repeatedly feeds an AI agent the same prompt until the task is complete. The agent sees its previous work through git history and modified files, enabling self-correction and iterative improvement.

"Ralph is a Bash loop." - Geoffrey Huntley, Creator of the Ralph Wiggum Technique

Why "Ralph Wiggum"?

Named after the lovably persistent character from The Simpsons, the technique embodies Ralph Wiggum's philosophy of determined persistence despite setbacks. Tech communities picked the name because of Ralph's nature to keep trying, no matter what - which perfectly describes how this AI loop operates.

How It Works

At its core, the Ralph Wiggum technique is elegantly simple: keep feeding an AI agent a task until the job is done. Where traditional agentic workflows stop when an LLM finishes calling tools, Ralph keeps going - verifying completion, providing feedback, and running another iteration until the task actually succeeds.

Key Mechanism: Stateless Resampling

Standard agent loops suffer from context accumulation - every failed attempt stays in the conversation history. After a few iterations, the model must process a long history of noise before focusing on the task. This leads to what's called "context rot."

Ralph Loop solves this by starting each iteration with fresh context. The prompt never changes, but the world does - files are modified, commits are made, and the agent reads these changes to understand progress.

The Core Insight: Git history becomes the AI's memory. Each commit is a checkpoint. The loop becomes a learning mechanism. This is why Ralph works for long-running tasks - the AI doesn't lose context because it builds on the filesystem state.

Methodology: Three Phases, Two Prompts, One Loop

The Ralph Wiggum technique operates as a structured funnel with distinct phases:

Phase 1: Define Requirements

Through LLM conversation, break project ideas into Jobs to Be Done (JTBD), then decompose these into topics of concern. Each topic generates a specification document.

Phase 2-3: Run Ralph Loop

Two operational modes share the same execution mechanism but use different prompts:

Planning Mode

Uses PROMPT_plan.md to perform gap analysis between specifications and existing code. Outputs a prioritized TODO list. No implementation, no commits.

Gap Analysis Read-Only Prioritization

Building Mode

Uses PROMPT_build.md. Assumes a plan exists, picks tasks, implements them, runs tests, and commits. This is where the actual coding happens.

Implementation Testing Commits

Two Modes of Operation

Mode HITL (Human-in-the-Loop) AFK (Away From Keyboard)
Description Resembles pair programming. You and the AI work together, reviewing code as it's created. Long-running, autonomous, unsupervised coding. Set it running and come back when done.
Iterations Single iterations with review between each Multiple iterations (10-50+) running continuously
Best For Learning Ralph, architectural decisions, risky integrations Mechanical refactors, well-defined tasks, overnight runs
Supervision Active steering and adjustment Minimal - notifications when finished

Important: Use HITL Ralph for early architectural decisions - the code from these tasks stays forever, and shortcuts cascade through the project. Save AFK Ralph for when the foundation is solid.

Implementation

The Simplest Form

At its most basic, Ralph is just this:

# The original Ralph loop - elegantly simple
while :; do
    cat PROMPT.md | claude
done

Enhanced Loop Script

A more production-ready implementation with mode selection and iteration limits:

#!/bin/bash
# loop.sh - Enhanced Ralph implementation

MODE=${1:-build}
MAX_ITERATIONS=${2:-0}
ITERATION=0

while :; do
    # Check iteration limit
    if [[ $MAX_ITERATIONS -gt 0 && $ITERATION -ge $MAX_ITERATIONS ]]; then
        echo "Max iterations reached ($MAX_ITERATIONS)"
        exit 0
    fi

    ((ITERATION++))
    echo "=== Iteration $ITERATION ==="

    # Select prompt based on mode
    if [[ "$MODE" == "plan" ]]; then
        PROMPT_FILE="PROMPT_plan.md"
    else
        PROMPT_FILE="PROMPT_build.md"
    fi

    # Run Claude with the prompt
    cat "$PROMPT_FILE" | claude \
        -p \
        --dangerously-skip-permissions \
        --output-format=stream-json \
        --model opus

    # Check for completion signal
    if grep -q "<promise>COMPLETE</promise>" output.log; then
        echo "Task completed successfully!"
        exit 0
    fi
done

Best Practices & Key Concepts

The Philosophy: Sit on the Loop, Not In It

Your job is to engineer the setup and environment that allows Ralph to succeed. Ralph should be doing all of the work, including deciding which planned work to implement next and how to implement it.

Backpressure Over Prescription

Don't prescribe how to implement things. Instead, create gates that reject bad work: tests, typechecks, lints, and builds that will fail on invalid output.

Observe and Tune

The prompts you start with won't be the prompts you end with. Watch for failure patterns and add "signs" to help Ralph avoid them next time.

Keep Tasks Small

Stories must fit within one context window. Oversized tasks cause incomplete implementations. Target specific changes, not entire features.

Keep CI Green

Each iteration must pass tests or you're in trouble. Broken code compounds across iterations and derails the entire loop.

Backpressure Mechanisms

Backpressure is the key steering mechanism. Create signals that reject invalid work:

  • Type checking - TypeScript, mypy, or similar static analysis
  • Linting - ESLint, Prettier, or code style enforcement
  • Unit tests - Fast, deterministic validation
  • Integration tests - End-to-end verification
  • Build process - Compilation must succeed
  • LLM-as-judge - For subjective criteria that need human-like judgment

Critical: Make sure the validation commands in AGENTS.md actually run and fail when they should. Claude only knows tests failed if the command returns a non-zero exit code.

Context Management

The "dumb zone" hits around 40% context utilization. Past that, reasoning degrades. Ralph's one-task-per-iteration structure sidesteps this by starting each iteration fresh.

Critical Language Patterns

Geoff Huntley discovered specific phrasing that improves effectiveness:

  • Use "study" rather than "read" or "examine"
  • Include "don't assume not implemented" to prevent redundant work
  • Specify subagent parallelization limits
  • Restrict build/test validation to single sequential subagent for backpressure control
  • Emphasize "capture the why" in documentation

Safety Considerations

Security Warning: The --dangerously-skip-permissions flag grants the AI full control over the terminal. Always run Ralph sessions in sandboxed environments (disposable cloud VMs, Docker containers, or services like Fly Sprites/E2B) to prevent accidental damage.

  • Always cap iterations for AFK mode (5-10 for small tasks, 30-50 for larger ones)
  • Use isolated environments with minimal necessary access
  • Restrict network connectivity where feasible
  • Never include credentials beyond task requirements
