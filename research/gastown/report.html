1. Overview - What is Gas Town?

Gas Town (also stylized as "Gastown") is a multi-agent orchestration system designed specifically for coordinating multiple AI coding agents, primarily Claude Code instances. Created by Steve Yegge and released in January 2026, it functions as a workspace manager that enables developers to run 20-30 parallel AI agents working on different tasks simultaneously.

Key Insight: Gas Town is described as "Kubernetes for AI agents" - instead of orchestrating containers, it orchestrates AI coding assistants to work in parallel on large codebases.

The name "Gas Town" is a reference to the oil refinery citadel from the Mad Max film series. This Mad Max theme continues throughout the framework with terminology like Mayor, Rigs, Polecats, the Refinery, and Convoys.

What Problem Does It Solve?

  • Context Loss: Traditional AI agents lose context when they restart. Gas Town persists work state in git-backed storage, enabling reliable multi-session workflows.
  • Parallelization: Instead of one agent working sequentially, Gas Town coordinates multiple agents working in parallel on different tasks.
  • Work Management: Provides a complete ticketing/issue tracking system designed specifically for AI agents to pick up, execute, and hand off work.

3. Architecture and How It Works

Gas Town uses an operational model rather than mimicking traditional software development hierarchies. The architecture consists of several key components:

Core Components

  • Town: The central workspace directory that contains all projects and agents
  • Rigs: Project containers that wrap Git repositories. Each Rig manages its own agents and work
  • Hooks: Git worktree-based persistent storage where work is assigned to agents
  • Beads: Atomic units of work stored as git-backed issues (the issue tracking substrate)
  • Convoys: Grouped work items that track delivery of features or bug fixes

The GUPP Principle

GUPP (Gas Town Universal Propulsion Principle) is the signature architectural principle: "When an agent finds work on their hook, they execute immediately. No confirmation. No questions. No waiting."

This prevents the "stalled system" failure mode where restarted agents wait for instructions instead of resuming work. It ensures the system keeps moving forward automatically.

4. The MEOW Stack (Molecular Expression of Work)

Gas Town operates on what Steve Yegge calls the MEOW Stack - a framework for representing work in a way that AI agents can execute reliably.

Key Concepts

Beads (Atoms)

Atomic units of work. Each bead is a git-backed issue with ID, description, status, and assignee. Format: gt-abc12

Molecules (Workflows)

Durable, sequenced workflows that survive crashes. Molecules can have complex shapes, loops, gates, and are Turing-complete.

Formulas

Pre-defined workflow templates that can be "cooked" and assigned to agents. Think of them as recipes for common development tasks.

Convoys

Special beads that wrap multiple work items into a trackable delivery unit - basically features or releases.

Why MEOW Matters: "Molecularized work" transforms ephemeral chat logs into persistent, git-backed objects. If a workflow is captured as a molecule, it survives agent crashes, compactions, restarts, and interruptions. Just start the agent up and it picks up where it left off.

5. Agent Roles

Gas Town defines seven distinct agent roles, each with specific responsibilities:

👑 The Mayor

Your primary AI coordinator. The Mayor is a Claude Code instance with full context about your workspace, projects, and agents. This is your main entry point - you tell the Mayor what to accomplish, and it coordinates everything else.

🛠 Polecats

Ephemeral per-rig workers that spin up on demand. They execute tasks in parallel, produce Merge Requests, then are decommissioned. Best for well-defined, well-specified work.

🔥 The Refinery

Manages the merge queue. An intelligent agent that understands backpressure and mediates between Polecats and the codebase, preventing chaotic collision of changes.

👁 The Witness

Team lead monitoring worker health and managing agent lifecycles. Watches over the system to ensure agents are functioning properly.

⚔ The Deacon

Handles system maintenance and cleanup tasks. Keeps the workspace healthy and performs administrative duties.

🐶 Dogs

Monitoring agents that watch for specific conditions and alert when issues arise. Part of the observability layer.

👥 Crew

Your personal workspace within a rig. The Crew is where you do hands-on development alongside the AI agents.

6. Workflow and Commands

Installation

# Install Gas Town via Go
go install github.com/steveyegge/gastown/cmd/gt@latest

# Add Go binaries to PATH
export PATH="$PATH:$HOME/go/bin"

# Create workspace with git initialization
gt install ~/gt --git

Prerequisites

  • Go 1.23+
  • Git 2.25+
  • Beads 0.44.0+
  • sqlite3
  • tmux 3.0+
  • Claude Code CLI (or compatible AI runtime)

Basic Setup

# Navigate to workspace
cd ~/gt

# Add your first project (rig)
gt rig add myproject https://github.com/you/repo.git

# Create your crew workspace
gt crew add yourname --rig myproject

# Start Gas Town daemon and Mayor
gt start

# Attach to Mayor session
gt mayor attach

Key Commands

Command Description
gt start Start daemon and Mayor session
gt mayor attach / gt may at Enter Mayor session
gt crew attach / gt crew at Enter Crew workspace
gt convoy list List active work across all rigs
gt convoy create "name" <issues> Create new convoy
gt sling <bead-id> <rig> Assign work to an agent
bd ready Show available work (Beads command)
bd formula list List available formulas
bd cook <formula> Execute a formula

Mayor Natural Language Commands

Once attached to the Mayor, you can use natural language:

  • Work Status: "show convoys" / "show status" / "what's blocked" / "what's ready"
  • Agent Management: "nudge polecats" / "restart all" / "clean up polecats"
  • Work Dispatch: "sling X to a polecat" / "create convoy for X" / "fix all four in parallel"

Gas Town vs Swarms Framework

While both handle multi-agent orchestration, they target different use cases:

  • Gas Town: Specifically designed for AI coding agents, Git-native, workspace-focused
  • Swarms: General-purpose enterprise framework, supports various agent network topologies

9. Considerations and Limitations

Strengths

  • Massive parallelization (20-30 agents)
  • Git-native persistence survives crashes
  • Well-defined agent roles and responsibilities
  • Molecularized work for reliable handoffs
  • Active community and rapid development
  • Support for multiple AI runtimes

Challenges

  • Cost: ~$100/hour in Claude tokens at peak
  • Complexity: Steep learning curve
  • Early Stage: Still evolving rapidly
  • Monitoring: Requires attention to track multiple agents
  • Safety: Runs in "unsafe" mode by default
Important Consideration: Gas Town is designed for a specific ambition level - massive parallelization across large codebases with automated issue triage, planning, and implementation at scale. If you're not operating at that level, the complexity may not be worth it. For smaller projects, simpler approaches may be more practical.

Who Should Use Gas Town?

  • Teams working on large codebases needing parallel development
  • Developers comfortable with complexity and early-stage tools
  • Organizations with budget for high token consumption
  • Those who have thought "I wish I had 100 Claude Codes"

Who Should Wait?

  • Solo developers on small projects
  • Those needing production-ready, battle-tested tools
  • Budget-constrained teams
  • Anyone wanting simple, single-agent workflows
